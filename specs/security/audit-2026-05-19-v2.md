## Critical findings cleared in v17.0.2

### F-CVE-01 — mysql2 RCE (CVSS 9.8)
- **CVE:** [GHSA-fpw7-j2hg-69v5](https://github.com/advisories/GHSA-fpw7-j2hg-69v5)
- **Title:** mysql2 Remote Code Execution via `readCodeFor` function
- **Affected:** mysql2 `<3.9.4` — v17.0.1 shipped with `mysql2@2.3.3`
- **Runtime exposure:** Yes — backend connects to MariaDB on every panel
  request. A malicious DB response could trigger the RCE; since MariaDB
  is local + FreePBX-controlled, the practical exploit path is limited
  but the CVE class is "remote".
- **Fix in v17.0.2:** `mysql2: ^2.3.3` → `^3.11.0`. mysql2 v3 API is
  backward-compatible for our usage (`createPool` + `query`).
- **Verified:** backend `npm audit` shows mysql2 critical resolved.

### F-CVE-02 — socket.io-parser injection (CVSS 9.8)
- **CVE:** [GHSA-qm95-pgcg-qqfq](https://github.com/advisories/GHSA-qm95-pgcg-qqfq)
- **Title:** Insufficient validation when decoding a Socket.IO packet
- **Affected:** socket.io-parser `>=4.0.0 <4.0.5` — v17.0.1 shipped
  with `socket.io@4.4.0` pulling parser `4.0.4`
- **Runtime exposure:** Yes — every WebSocket message goes through the
  parser. A malicious client could send a crafted packet to trigger
  unbounded resource consumption or potential injection.
- **Fix in v17.0.2:** `socket.io: ^4.4.0` → `^4.8.0`. Pulls in
  parser ≥ 4.2.5 which has the fix.
- **Verified:** backend `npm audit` shows socket.io-parser critical
  resolved.

### F-CVE-03 — frontend socket.io-client → socket.io-parser injection
- Same CVE as F-CVE-02 but in the frontend
- **Fix in v17.0.2:** `socket.io-client: ^4.4.1` → `^4.8.0`
- **Verified:** frontend `npm audit` shows it resolved.

## High findings remaining (deferred)

Backend `npm audit` still shows 6 high CVEs, all in dev/build
toolchain transitive deps:

- `@babel/traverse <7.23.2` (RCE in compile path — only an attack
  surface during the build step, NOT at runtime)
- `cross-spawn <7.0.6` (CRA dev server)
- `nanoid <3.3.8` (jest internal)
- `webpack <5.94.0` (CRA build)
- etc.

These don't ship in the runtime artifact. The build artifact (PM2's
`build/src/main.js` for backend, `frontend/build/static/js/*.js` for
frontend) does NOT include these vulnerable code paths. They're only
present in `node_modules/` during install.

**However:** an attacker who could trick the install process into
fetching a malicious package version could exploit them. The
mitigation is `package-lock.json` (which we ship + git-track),
pinning every transitive version. As long as users install via
`npm ci` (which we do in the install method) the lockfile is
honored.

**Deferral rationale:** clearing these requires bumping the build
toolchain (jest 27 → 29, ts-jest 27 → 29, react-scripts 5 → Vite,
eslint 8 → 9). This is a multi-day modernization push tracked
under "Phase 05 — modernization" (not yet started).

## Code-level review

### SQL injection — ✅ safe
All DB queries use `mysql2/promise` parameterized queries:

- `src/api/auth.ts:5` — `getDb().query('SELECT ... WHERE username = ?', [username])`
- `src/api/auth.ts:13` — same pattern
- `src/api/auth.ts:22` — parameterized subquery
- `src/func/call-logs.ts` — parameterized
- `src/func/caller-id.ts` — parameterized

No string concatenation into queries anywhere. ✅

### XSS — ✅ safe
- Frontend uses React's JSX auto-escaping for all user-provided text
- No `dangerouslySetInnerHTML` calls
- Phonebook XML generators (`phonebook-yealink.ts`, `phonebook-fanvil.ts`)
  use template literals with values from the FreePBX-managed
  `contactmanager_*` tables, NOT from HTTP request inputs. XML
  injection risk is limited to whatever FreePBX itself stores.

### CSRF — ⚠️ partial
- The FreePBX admin page (`views/main.php`) implements its own CSRF
  via `randcheck` (random session token); the `doConfigPageInit` now
  uses strict (`!==`) comparison (fixed in v17.0.0). ✅
- The NodeJS backend's WebSocket API uses Socket.IO auth handshake
  with username/password — not classical CSRF tokens. Since
  WebSockets are typically not subject to CSRF the way XHR is, this
  is acceptable, BUT the panel's HTTP REST endpoints
  (`/lookupcallerid`, `/fanvil-phonebook.xml`, etc.) are GET-only
  and read-only, so CSRF doesn't apply to them either.
- **Verdict:** no exposed state-changing HTTP endpoint without a token.

### Authentication — ✅ correctly delegates to FreePBX
- See the F-03 correction in
  [audit-2026-05-19.md](audit-2026-05-19.md). Auth uses FreePBX User
  Manager with per-user bcrypt-verified passwords + module permission
  check.

### Authorization — ⚠️ coarse-grained
- A logged-in user with panel access sees ALL calls and ALL contacts.
  No per-user filtering. For a typical home/small-business deployment
  this is acceptable (everyone authorized is staff). For a
  multi-tenant deployment, this is a gap.
- **Documented as known limitation** in [../../docs/FAQ.md](../../docs/FAQ.md).

### Secrets in code/logs — ✅ no leaks
- No hardcoded credentials in source
- Logs don't print passwords (verified in `auth.ts` — only username
  appears on connection)
- The AMI password from `manager.conf` is held in memory only

### Path traversal — ✅ no user input in file paths
- The only file reads against user-influenced paths are:
  - `getConfig().freepbxConfFile` — set in config.local.json (admin
    only), defaults to `/etc/freepbx.conf`
  - `getConfig().managerConfFile` — same
- No `req.params` or `req.query` flows into `fs.readFile` calls.

### Error information leakage — ⚠️ minor
- Panel errors are logged with full stack traces to PM2 stdout/stderr
- Frontend doesn't expose stack traces to logged-in users
- HTTP API endpoints don't echo internal errors back to clients
- **Verdict:** logs are admin-only (require root/asterisk to read).
  Acceptable.

## Network exposure

| Endpoint | Default bind | Authenticated | Encrypted by default |
|---|---|---|---|
| `:4848/callpanel/` (static) | `0.0.0.0` | No (public static assets) | No |
| `:4848/callpanel/lookupcallerid` | `0.0.0.0` | No | No |
| `:4848/callpanel/fanvil-phonebook.xml` | `0.0.0.0` | No | No |
| `:4848/callpanel/yealink-phonebook.xml` | `0.0.0.0` | No | No |
| `:4848/callpanel/socket.io` | `0.0.0.0` | **Yes** (FreePBX User Manager) | No |

**Risk:** the phonebook XML endpoints expose all your contacts
without authentication to anyone who can reach port 4848. In a
typical home/small-business deployment behind NAT this is fine;
on a public-IP PBX it's a data leak.

**Mitigations:**
1. **Firewall block port 4848 inbound** (documented in
   [../../docs/CONFIGURATION.md](../../docs/CONFIGURATION.md))
2. **Reverse-proxy via Apache** so port 4848 stays local-only and
   the phonebook XMLs go through whatever auth + IP filtering you
   layered on Apache (also documented)

**Tracked for future hardening:** bind the backend HTTP server to
`127.0.0.1:4848` by default and require the reverse-proxy path.
This is a breaking change for users running without the proxy, so
it should ship in a v18.x major.

## Honest gaps in this audit

- **No penetration testing.** No active exploitation attempt has been
  made against a deployed instance.
- **No fuzzing of the WebSocket protocol.** Socket.IO's parser is
  upstream-maintained; relies on their security work.
- **No FIPS / formal cryptography review.** Uses standard bcrypt
  (cost factor inherited from FreePBX User Manager).
- **No supply-chain attestation.** Builds aren't signed; release
  tarballs don't carry SLSA provenance. (For a stronger guarantee,
  see the npm Trusted Publishing playbook at
  `~/.claude/npm-trusted-publishing-playbook.md` — but this isn't
  an npm-published package, it's a FreePBX module tarball.)

## Audit-cadence recommendation

- **`npm audit`** before every release. Trivial — `npm audit --json`
  exits non-zero on any finding.
- **SonarQube scan** at least quarterly, and after any large code
  change. The scanner CLI is checked in at
  `~/Documents/GITprojects/TicketsCADFixes/sonar-scanner-temp/`.
- **Manual code review** when touching auth, DB queries, or anything
  that handles user-provided input.
- **Dependency major-version review** annually — even when there are
  no CVEs, EOL'd majors stop getting security patches.
