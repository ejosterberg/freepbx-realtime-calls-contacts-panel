# Security audit — 2026-05-19

Initial security review of the fork, conducted alongside the FreePBX 16/17
baseline validation work.

## Scope

- `Callpanel.class.php` — PHP module wrapper
- `views/main.php` — admin config UI
- `calls-contacts-panel/src/` — NodeJS backend (TypeScript)
- `calls-contacts-panel/frontend/src/` — React frontend (TypeScript)

Not in scope (this audit): full SonarQube scan, dependency CVE scan,
deep crypto review.

## Findings

### F-01 — `$_POST` / `$_SESSION` / `$_SERVER` null-key access (PHP 8.x deprecation)
**Severity:** Low (DoS via repeated 500s only; no data leak)
**Status:** Fixed in this fork.
**Location:** `Callpanel.class.php:105, 177, 199`; `views/main.php:3, 97, 104, 111`
**Fix:** Added `?? null` / `?? ''` defaults. PHP 7.4 silently returned NULL;
PHP 8.x raises `Undefined array key` → `E_DEPRECATED` (and `Whoops`
intercepts those on FreePBX 17, blocking the page).

### F-02 — CSRF token comparison missing strict-equal
**Severity:** Low (originally `!=` instead of `!==`)
**Status:** Fixed.
**Location:** `Callpanel.class.php:105` (now `!==`)
**Rationale:** PHP loose comparison can match `0 == "foo"` in edge cases.
The form's `randcheck` was `rand()` → integer-stringy; using `!==`
prevents any unintended loose match.

### F-03 — Backend auth uses single shared bcrypt password
**Severity:** Medium
**Status:** Documented as known limitation (not changed in this release).
**Location:** `calls-contacts-panel/src/api/auth.ts`
**Rationale:** The panel ships with a single admin password rather than
per-user auth. This is acceptable for the intended use case (small
office) but should be flagged. Future work: integrate FreePBX's
User Manager (`userman` module) for per-user auth.

### F-04 — CORS configured as `origin: true, credentials: true`
**Severity:** Medium (allows any origin to send credentialed requests)
**Status:** Documented as known limitation.
**Location:** `calls-contacts-panel/src/web-server.ts:34-37`
**Rationale:** This combination is overly permissive. Future work:
restrict origin to the FreePBX host's URL when behind a reverse proxy.

### F-05 — Backend HTTP listens on `0.0.0.0:4848` (not 127.0.0.1)
**Severity:** Medium (panel reachable from outside the LAN if firewall is permissive)
**Status:** Documented; mitigation = run behind Apache reverse proxy.
**Location:** `calls-contacts-panel/src/web-server.ts:24` (via Express default)
**Recommendation:** README documents the Apache reverse-proxy approach.
On a public-facing FreePBX, the firewall must block port 4848 inbound.

### F-06 — `exec()` calls in install path use shell-escaped paths
**Severity:** Informational
**Status:** OK in this fork (added `escapeshellarg()`).
**Location:** `Callpanel.class.php` install method (this fork's additions)
**Rationale:** New `exec()` calls for `npm run build` use `escapeshellarg()`
on the directory path. Upstream's `exec('rm -Rf ' . $this->nodeloc . '/node_modules')`
in `uninstall()` is unchanged — `$this->nodeloc` is set in the constructor
from `__DIR__`, not user input, so this is safe.

### F-07 — `cors: '*'` not actually used; `cors({origin:true})` is reflective
**Severity:** Informational (clarification on F-04)
**Status:** —
**Rationale:** `origin: true` in the cors package means "reflect the
Origin header" — equivalent to `*` for browsers that don't send Origin,
but actually returns the requesting origin in `Access-Control-Allow-Origin`.
Combined with `credentials: true`, this is the unsafe pattern. Fix in
F-04 future work.

### F-08 — Default config has no admin password
**Severity:** High if deployed without setup; Low if installer requires it.
**Status:** Needs verification on first install (TBD in phase-01 testing)
**Location:** `calls-contacts-panel/config.default.json`
**Recommendation:** During phase-01 install, verify whether the install
flow forces an admin password to be set. If not, README must call this
out as a required post-install step.

## Cleared items (not vulnerabilities)

- SQL queries in `calls-contacts-panel/src/func/*.ts` use parameterized
  `mysql2` queries throughout. No string concatenation in WHERE clauses.
- Frontend output uses React's automatic JSX escaping; no `dangerouslySetInnerHTML`.
- Phonebook XML generators use template literals, but values come from
  the FreePBX-managed ContactManager DB (not user input directly from
  HTTP), so XML injection risk is limited to ContactManager itself.
